This is my first ever room that I have published on TryHackMe. It's a simple challenge but it might be hard to figure out where to start for some individuals.

## Writeup

So initially, when you open the website, you are greeted with a login page and a link to the registration page. 

![login.png](https://raw.githubusercontent.com/AbdulRKB/img/refs/heads/main/bitnotes/1.png)

Upon registering and logging in, you'd be redirected to the dashboard page, which shows a list of saved notes and has the ability to add new notes. 

![dashboard.png](https://raw.githubusercontent.com/AbdulRKB/img/refs/heads/main/bitnotes/2.png)


The dashboard page is pretty much a red herring, there's nothing of interest here, the page provides a basic functionality of adding notes and looking at the saved notes. 

The intended vulnerability is in the `Account` page.

![accounts.png](https://raw.githubusercontent.com/AbdulRKB/img/refs/heads/main/bitnotes/3.png)

Although there's no obvious user input in this page, as in, there are no buttons or input textboxes - there is something really interesting in the page that is user-controlled: user agents.

When the user logs in, the web server grabs the user agent from the http request, and saves it in the database, and whenever the `/account` page is visited, it displays user agents associated with the specific user who's currently logged in. 

The key fact about user agents is that they're client-controlled, you can easily change the user agents to your liking, and there's many ways to do it, you can use:
- Burp Suite
- Browser Extension
- Curl
- Python Requests

As a demonstration, I created a custom Python script to show this

```python
import requests

headers = {
    'User-Agent': "haha this is my custom user agent :)",
}

data = {
    'email': 'admin@hack.thm',
    'password': 'admin',
}

response = requests.post('http://10.128.181.232:5000/login', headers=headers, data=data)

print(response.status_code)
```


![custom-agent.png](https://raw.githubusercontent.com/AbdulRKB/img/refs/heads/main/bitnotes/4.png)


Some might've also thought this is an XSS challenge, and might have added a statement such as `<script>alert(0)</script>` as the user agent. This would in fact be a partially valid assumption as the page is also vulnerable to XSS attacks, but that isn't the path to get the flag.

![xss.png](https://raw.githubusercontent.com/AbdulRKB/img/refs/heads/main/bitnotes/5.png)


The page is vulnerable to [SSTI Attacks](https://portswigger.net/web-security/server-side-template-injection). You can use the polyglot `${{<%[%'"}}%\.` as your user agent to test if the web app is vulnerable to SSTI Attacks, as it would lead to an error (You'd have to reset the machine to resolve it). 

Once it has been established that the web app is vulnerable to ssti attacks, the next logical decision would be to find out what template engine the web app uses.

One can easily see that the web app is running on a Python server by going to `DevTools -> Network -> Packet -> Headers -> Response Headers -> Server`. This narrows down the search, for Python webapps, Jinja is widely used as the go-to template engine. 

![server-details.png](https://raw.githubusercontent.com/AbdulRKB/img/refs/heads/main/bitnotes/6.png)

You could verify that Jinja is being used by setting the user agent as `7*7` and it'd lead to the web app showing the number `49` on the `/account` page.

Then, to run remote commands on the web server, you can look up "Jinja SSTI". The article by _OnSecurity_ is awesome: [Server Side Template Injection with Jinja2 ](https://onsecurity.io/article/server-side-template-injection-with-jinja2/)


After reading the article, you'd know what to use as your user agent to read the flag. You can use 

```python
{{request.application.__globals__.__builtins__.__import__('os').popen('ls').read()}}
```

to list the web servers directory. And use

```python
{{request.application.__globals__.__builtins__.__import__('os').popen('cat flag.txt').read()}}
```

to see the flag.
